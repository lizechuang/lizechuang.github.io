基于plist注册的WebView跳转路由

前言

前阵子在写项目中有一个需求，显示一些Html界面，后台返回html数据由我们这边显示，我们是直接用WKWebview去显示这个界面，并把包在tableViewCell中去显示。html数据中有些标签是有点击跳转到显示标签具体数据页面的业务，（标签后面带有具体数据请求URL的absoluteString）

这个时候可以有暴力的方法当点击的时候直接获取到URL的absoluteString数据然后直接用当前的navigationC去push到具体的页面VC去显示，这样确实Ojbk，但是在cell的回调方法中需要去解析这个absoluteString，然后再去判断要跳转到哪个页面。而且，假如我们当前的navigationC在第三个tab中，跳转的页面要去第二个tab，此时要怎么做，这个时候好像可以去获取tabbarVC再去调整，这个时候再回头去看当前VC的代理方法中，你会发现我怎么在这个代理方法中写了这么多判断，把tabbarVC和隔壁的navigationC耦合到我们这个VC，这种做法是不可取的。

还有一个很傻逼的地方，如果我们在需要用到WKWebView点击标签跳转的地方都要加上这些判断，很浪费时间，倒不如把这些跳转整合到一个路由中，将当前的absoluteString输入进去，怎么去跳转由路由去决定。

Effective Objective-C

使用消息结构的语言，其运行时所应执行的代码由运行环境来决定；

使用函数调用的语言，则由编译器决定；

对象所占内存总是分配在“堆空间”中，而绝不会分配在“栈”上。

堆空间存放对象，栈帧存放指向对象的指针.

在类的头文件中尽量少引入其他头文件

多使用字面量语法，少用与之等量的方法

多用类型常量，少用#define预处理指令

ps:若常量局限于某“编译单元”（也就是“实现文件”）之内，则在前面家字母k；若常量在类之外可见，则通常以类名为前缀

static const 来定义只在编译单元内可见的常量

extern const 来定义全局常量

用枚举表示状态、选项、状态码

关于多个网络请求发送请求的设计

前言

我们在项目中经常会遇到并行发送多个网络请求或者串行发送多个网络请求的业务需求，这个时候我们要怎么去实现这个业务需求。

例如并行发起请求A，请求B，请求C，需要等到A，B，C都返回了才进行回调。我们可以像发送单个请求一样，在A，B，C的回调里面加上一层傻逼判断其他两个请求是否发送成功，做到同时发送了这三个请求,但是你会不会觉得这样会很奇怪，需要在业务层加上一层判断，逻辑就不对了。

又例如串行发起请求A，请求B，请求C，需要等到A(成功)->B(成功)->C(成功)才进行进行回调。我们可以利用Block嵌套A回调嵌套B，B回调嵌套C。好像是可以实现，但是不觉得代码会很奇怪吗，很多层嵌套，如果又有一个请求，又加入这个嵌套吗？这种做法十分肤浅。
